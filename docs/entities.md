# Entities, events, and services

Entity IDs are generated by Home Assistant from installation and device names.
The examples below use placeholders; select the actual entity from Home
Assistant's entity picker instead of copying an example ID literally.

## Entities

| Platform | Entity | Behavior |
| --- | --- | --- |
| `alarm_control_panel` | Verisure Alarm | Reports disarmed, armed-home, armed-away, and pending states; supports arm-home, arm-away, and disarm commands. |
| `binary_sensor` | Door/window | Reports the cloud-provided open/closed state and last report time. |
| `binary_sensor` | Ethernet status | Reports gateway broadband connectivity. |
| `binary_sensor` | Guardian | Reports whether the Guardian feature is activated for the installation. |
| `camera` | Smart camera | Displays the latest available image and supports requesting a new capture. |
| `lock` | Smart lock | Reports lock state and supports lock, unlock, and autolock services. |
| `sensor` | Temperature/humidity | Reports climate values exposed by installed Verisure sensors. |
| `sensor` | Remaining SMS | Reports the SMS balance; boolean attributes indicate which operations consume SMS. |
| `switch` | Smart plug | Reports and controls a smart plug. |
| `update` | Gateway firmware | Read-only installed/latest firmware status. Installation is not exposed. |
| `device_tracker` | Verisure user | Optional, disabled by default, and only created when tracking is enabled in integration options. |

Entities only appear when the selected installation exposes the corresponding
device or capability. Firmware entities are read-only even if Home Assistant
shows that a newer version exists.

## Event entities

Event entities change state to a timestamp whenever a new matching GraphQL
event-log record is observed. Existing records from the first successful fetch
form a baseline and do not fire automations.

### Security alarm event

Supported `event_type` values:

- `intrusion`
- `fire`
- `sos`
- `water`
- `animal`
- `technical`
- `warning`

### Alarm activity event

Supported `event_type` values:

- `arm`
- `disarm`
- `lock`
- `unlock`
- `picture`

### Event attributes

| Attribute | Meaning |
| --- | --- |
| `event_type` | Normalized Home Assistant event type listed above. |
| `event_id` | Cloud event identifier used for in-memory deduplication. |
| `event_time` | Timestamp reported by Verisure. |
| `event_category` | Original GraphQL event category. |
| `verisure_event_type` | Verisure's more specific event subtype, when supplied. |
| `event_source` | Source type reported by Verisure. |
| `arm_state` | Related alarm state, when supplied. |
| `area` | Related area, when supplied. |
| `device_label` | Related Verisure device identifier, when supplied. |

Contact/user names are intentionally omitted from event attributes. Event IDs,
times, areas, and device identifiers are redacted from downloaded diagnostics.

The API returns at most 15 recent records for each of the security and activity
groups per poll. A very high event volume between polls can therefore displace
older records. Events that happened while Home Assistant was stopped are used
as the new startup baseline and are not replayed.

## Entity services

All services require an entity target selected through Home Assistant.

| Service | Target | Effect |
| --- | --- | --- |
| `swe_verisure.capture_smartcam` | Camera | Requests a new SmartCam image and waits for availability. |
| `swe_verisure.enable_autolock` | Lock | Enables autolock for the selected smart lock. |
| `swe_verisure.disable_autolock` | Lock | Disables autolock for the selected smart lock. |

Alarm, lock, and smart-plug controls use the normal Home Assistant entity
actions. Cloud acceptance and physical completion can take several seconds.
